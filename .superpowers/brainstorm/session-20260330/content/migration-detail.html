Section 2: High-Impact Migration (Home & Player)

Solving state complexity and oversized composables through MVI

Target: VideoPlayerScreen.kt

The Problem (Current)

  • Scattered StateFlow for: position, buffer, quality, subtitles, transcoding status.
  • Manual synchronization between UI actions and playback state.
  • Composables (>1000 lines) pass down many lambdas for state updates.

The Solution (MVI)

  • Single Source: VideoPlayerUiState contains all playback and overlay flags.
  • Deterministic Actions: All UI inputs (seek, toggle, etc.) go through a single intent { ... } flow.
  • SideEffects: Navigation-to-Home or Errors are one-time events.
MVI State Flow Example
data class PlayerState(
  val isPlaying: Boolean = false,
  val currentPosition: Long = 0L,
  val transcodingStatus: TranscodingStatus = Idle
)

sealed class PlayerIntent {
  object TogglePlay : PlayerIntent()
  data class SeekTo(val pos: Long) : PlayerIntent()
}

// UI simplifies to:
val state by viewModel.collectAsState()
VideoControls(state, onIntent = viewModel::handleIntent)

Addressing "Large Composables" (Phase 3 Evolution)

The MVI transition acts as a forced refactor for your 1000+ line composables:

  • Stateless Sub-components: Large screens will be broken into smaller, stateless components (e.g., PlaybackOverlay.kt, QualitySelector.kt) that only accept UiState and an (Intent) -> Unit callback.
  • Improved Testability: We can test the entire state-machine logic in isolation from the Compose UI.
  • Haptics & Rich Feedback: MVI side-effects are the perfect place to trigger the "Rich Haptics" from your Android 16 plan.
